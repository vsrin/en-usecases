Pre-read · ElevateNow NL2SQL Architecture

NL2SQL fails not because language models cannot write SQL, but because they receive raw schema with no vocabulary bridge, no scope reduction, and no rule injection.

Every enterprise NL2SQL deployment confronts the same failure pattern: the model hallucinates column names, silently drops required filters, infers wrong join paths, and surfaces restricted fields. These are not model failures. They are architecture failures. This document presents a three-layer approach that reduces complexity before it reaches the model, not after.

01 Executive summary One page · Four takeaways

Four structural observations explain why NL2SQL keeps failing in production, and why the fix is upstream of the model, not downstream.

Each of the four summarizes a section of this pre-read. If only one page is read, this is the one.

I.

The root cause is raw schema reaching the model with no mediation layer.

Passing raw column names, raw data types, and raw questions to an LLM produces wrong column references, hallucinated joins, missing filters, and surfaced PII. Every error is predictable and preventable.

II.

Three layers, each doing one job, prune the problem before the model sees it.

The Auto Semantic Layer bridges vocabulary. The Domain Tuning Layer encodes business rules. The Query Pipeline reduces scope. Together they ensure that by the time the prompt is written, the problem is already solved.

III.

A governed semantic layer is a set of thirteen artifacts that replace raw schema entirely.

DDL Cache, Value Index, Business Rules, Join Keys, Hidden Fields, FAISS-indexed gold queries, and seven other objects form a complete semantic contract. The model never sees raw schema again.

IV.

The lifecycle from ingestion to production is a repeatable four-stage pipeline.

Ingest a new dataset, generate the semantic layer, tune via a lightweight interface, freeze and deploy to production. If scope changes post-deployment, the lifecycle repeats from ingestion.

03 The three layers Exhibit 2 of 5

Three layers, each doing exactly one job, reduce complexity before it reaches the model, and each prevents a distinct class of failure.

The layers are MECE: vocabulary bridging, rule injection, and scope reduction. Together they ensure the model receives only what it needs to answer reliably.

Exhibit 2 · Three pre-generation layers: what each does and what each prevents
So what. Each layer addresses one class of failure, and the three classes are MECE. Layer 1 fixes what the model calls things. Layer 2 fixes how the model applies business logic. Layer 3 fixes what the model is allowed to see. Remove any one layer and its corresponding failure class returns.

04 Ingestion pipeline Exhibit 3 of 5

The ingestion pipeline converts a raw dataset and a set of gold queries into thirteen semantic artifacts that replace raw schema entirely.

Two inputs (Data Scope and Gold Queries) feed parallel processing paths. The outputs form the complete semantic layer: the model never touches the original schema.

Exhibit 3 · Ingestion pipeline: from raw schema to governed semantic layer
Read this as. The semantic layer is not a configuration file. It is a set of thirteen interrelated artifacts that encode vocabulary (aliases, DDL cache), business logic (business rules, value index), access control (hidden fields), query patterns (FAISS index, chunks), and relationship definitions (join keys). All thirteen are produced by a single ingestion run from two inputs. All thirteen are loaded at generation time. The model never sees anything else.
05 Development lifecycle Exhibit 4 of 5

The lifecycle from raw dataset to production deployment is a repeatable four-stage pipeline, and scope changes re-trigger it from the beginning.

Ingest, build the semantic layer, tune via a lightweight interface, freeze and deploy. If scope changes post-deployment, the lifecycle repeats from ingestion.

Exhibit 4 · NL2SQL development lifecycle: ingestion, tuning, deployment
The commitment. Every new dataset gets one ingestion run that produces all thirteen semantic artifacts. The tuning stage is codeless: domain experts adjust business rules, add gold queries, and mark hidden fields through a GUI. When accuracy reaches the target standard, the semantic layer is frozen and deployed. No custom code, no model fine-tuning, no prompt engineering in production.
06 Query pipeline Exhibit 5 of 5

At query time, two operations transform the raw user question before the model sees it: EXPAND adds business context, and SCHEMA LINK selects only the columns needed to answer.

Together these two operations eliminate context loss across multi-turn conversations and prevent the model from being distracted by irrelevant tables and columns.

Exhibit 5 · Query-time pipeline: EXPAND and SCHEMA LINK
Upshot. By the time the language model receives its prompt, the question has been rewritten with full business context, the schema has been reduced to only relevant columns, join keys have been explicitly defined, hidden fields have been excluded, and similar gold queries have been retrieved as few-shot examples. There is nothing left to hallucinate.
07 Working session agenda Six items · 45 minutes

The goal of our meeting is a working session, not a demo.

This pre-read provides the architectural frame. The session itself should test the approach against your data, your vocabulary, and your accuracy requirements.

  1. 01

    Current NL2SQL failure patterns

    Which of the five failure modes (vocabulary mismatch, hallucinated columns, wrong metrics, hallucinated joins, missing filters) is most acute in your current deployment?

  2. 02

    Data scope and gold query inventory

    Identify the initial dataset scope and existing validated Q+SQL pairs that would seed the first ingestion run.

  3. 03

    Business rules and metric definitions

    Walk through the domain-specific metric definitions, date logic, and acronym disambiguation that the Domain Tuning Layer must encode.

  4. 04

    Privacy and hidden field requirements

    Confirm which fields must be masked, which population filters are mandatory, and how access control rules should be enforced.

  5. 05

    Accuracy targets and tuning workflow

    Define the accuracy standard that triggers the semantic layer freeze and the team that will own the tuning GUI.

  6. 06

    Open questions

    The questions your team is already asking about governed NL2SQL. We want to hear them before we propose a deployment path.
